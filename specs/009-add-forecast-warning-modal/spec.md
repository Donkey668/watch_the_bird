# Feature Specification: 天气与观鸟指数预报预警弹窗

**Feature Branch**: `009-add-forecast-warning-modal`  
**Created**: 2026-04-03  
**Status**: Draft  
**Input**: User description: "在“天气与观鸟指数”组件最底部的“请求时间”和“刷新结果”按钮下方增加一个深绿色按钮，点击后弹出顶层可滚动弹窗，按公园所在区县展示分区逐时预报、分区预报、日月时刻、灾害预警四个独立模块，并遵循指定字段与展示规则。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 从天气面板打开预报预警弹窗 (Priority: P1)

作为在“分析”页面查看天气与观鸟指数的用户，我希望在“请求时间”和“刷新结果”下方看到一个醒目的入口按钮，并在点击后打开顶层弹窗查看当前公园所在区县的预报预警信息，这样我不用离开当前页面就能继续判断是否适合外出观鸟。

**Why this priority**: 按当前公园快速获取预报预警是本功能主价值，没有入口按钮和弹窗容器，其余四个模块都无法触达用户。

**Independent Test**: 打开“分析”页面，在“天气与观鸟指数”组件底部确认存在“点击获取预报预警”按钮；点击后出现顶层弹窗且可纵向滚动，弹窗内容默认绑定当前地图所选公园所属区县。

**Acceptance Scenarios**:

1. **Given** 用户已进入“分析”页面并看到“天气与观鸟指数”组件，**When** 页面完成渲染，**Then** “请求时间/刷新结果”行下方显示一个深绿色圆角矩形按钮，按钮文案为“点击获取预报预警”，字体为居中白色文本。
2. **Given** 用户点击“点击获取预报预警”按钮，**When** 弹窗打开，**Then** 弹窗位于当前界面的图层顶层，弹窗内容可纵向滚动，并显示四个独立模块标题。
3. **Given** 用户在“公园地图”中切换了公园，**When** 再次打开弹窗或触发弹窗数据刷新，**Then** 弹窗数据上下文必须切换为新公园所在区县，不得继续展示旧区县结果。

---

### User Story 2 - 浏览分区逐时预报与分区预报卡片 (Priority: P2)

作为需要查看短时与近日天气变化的用户，我希望“分区逐时预报”和“分区预报”各自独立展示为圆角容器，并通过横向卡片快速比较不同预报时次的天气状态与温度，这样我能快速评估观鸟窗口期。

**Why this priority**: 逐时与分区预报是用户做当日/近期出行决策的核心信息，优先级仅次于弹窗入口本身。

**Independent Test**: 打开弹窗后检查两个模块均存在独立外层圆角容器；逐时预报仅展示当前北京时间之后的已更新时次，分区预报仅展示今天及之后时次；当时次数量过多时模块使用横向滚动卡片。

**Acceptance Scenarios**:

1. **Given** “分区逐时预报”返回多条记录，**When** 模块渲染，**Then** 每条记录使用独立圆角卡片展示三行内容并按顺序居中显示 `WEATHERSTATUS`、`QPFTEMP`、`DDATETIME`。
2. **Given** “分区逐时预报”数据包含早于当前北京时间的记录，**When** 模块筛选数据，**Then** 仅展示当前北京时间之后且已更新的记录。
3. **Given** “分区预报”返回多条记录，**When** 模块渲染，**Then** 每条记录使用独立圆角卡片展示三行内容并按顺序居中显示 `WEATHERSTATUS`、`MINTEMPERATURE/MAXTEMPERATURE`、`DDATETIME`。
4. **Given** 任一模块卡片数量超过可视宽度，**When** 用户浏览该模块，**Then** 模块内部提供横向滚动列表而非压缩文本或换行错乱。

---

### User Story 3 - 查看日月时刻与灾害预警明细 (Priority: P3)

作为出行前需要规避风险的用户，我希望在同一弹窗中查看当日日月时刻和当前有效灾害预警，并可以点开预警行查看详细内容和发布区县，这样我能在行程安排前完成安全确认。

**Why this priority**: 日月时刻和灾害预警是增强决策信息，重要但依赖前两类数据先可访问，因此设为 P3。

**Independent Test**: 打开弹窗后检查“日月时刻”模块按行展示 `ATTRIBNAME/ATTRIBVALUE` 且有分隔线；“灾害预警”模块仅显示当前北京时间仍处于“发布”且未取消的记录，点击预警行能查看 `ISSUECONTENT` 与 `DISTRICT` 明细。

**Acceptance Scenarios**:

1. **Given** “日月时刻”返回今天的多个属性，**When** 模块渲染，**Then** 每个 `ATTRIBNAME` 与 `ATTRIBVALUE` 独立成行，且相邻行之间使用分隔线分开。
2. **Given** “灾害预警”返回多个有效记录，**When** 模块渲染，**Then** 按返回的 `SIGNALTYPE` 顺序纵向逐行展示，行文案格式为“序号 SIGNALTYPE SIGNALLEVEL预警”，且 `SIGNALTYPE` 与 `SIGNALLEVEL` 之间不插入空格。
3. **Given** 某条预警 `SIGNALTYPE` 存在但 `SIGNALLEVEL` 为空，**When** 模块渲染该行文本，**Then** 该行按黄色文本规则展示并仍可点击查看详情。
4. **Given** 某条预警的 `SIGNALTYPE` 与 `SIGNALLEVEL` 均为空，**When** 模块渲染，**Then** 仅显示 `gray-400` 文本“当前无生效信号。”并不展示可点击预警详情。
5. **Given** 用户点击任意预警行文本，**When** 详情弹窗打开，**Then** 必须展示该记录的 `ISSUECONTENT` 与 `DISTRICT`。

### Edge Cases

- “分区逐时预报”或“分区预报”返回空数组时，模块如何保持容器结构并显示明确空状态提示而非空白区域？
- 返回记录中的 `DDATETIME` 格式异常或缺失时，模块如何避免错误排序并标记该条数据不可用？
- 四个模块中某一模块请求失败时，如何保证其余模块仍能展示，不因单点失败导致整个弹窗不可用？
- 用户在弹窗打开期间切换公园时，如何避免短暂显示旧区县与新区县数据混杂？
- 灾害预警同时存在“发布”和“取消”状态记录时，如何确保最终只保留当前北京时间下仍有效的“发布”记录？
- 预警行颜色映射缺少对应级别定义时，如何保证回退为可读文本颜色并不影响详情查看？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在“天气与观鸟指数”组件底部“请求时间/刷新结果”行下方新增一个深绿色圆角矩形按钮，按钮文案为“点击获取预报预警”，并保持白色居中文本。
- **FR-002**: 用户 MUST 能通过点击该按钮打开一个位于图层顶层的弹窗，且弹窗内容区支持纵向滚动。
- **FR-003**: 弹窗数据上下文 MUST 与“公园地图”当前选中公园的区县保持一致。
- **FR-004**: 弹窗 MUST 包含并显示四个彼此独立的数据模块：分区逐时预报、分区预报、日月时刻、灾害预警。
- **FR-005**: 系统 MUST 为每个模块提供独立的加载、空状态和错误状态展示，不得因单模块异常阻断其他模块展示。
- **FR-006**: “分区逐时预报”模块 MUST 使用外层圆角矩形容器包裹。
- **FR-007**: “分区逐时预报”模块 MUST 按 `DDATETIME` 筛选出当前北京时间之后且已更新的记录。
- **FR-008**: “分区逐时预报”模块中每个预报时次 MUST 以独立圆角卡片展示三行信息，并按顺序居中显示 `WEATHERSTATUS`、`QPFTEMP`、`DDATETIME`。
- **FR-009**: 当“分区逐时预报”卡片数量超过可视宽度时，模块 MUST 使用横向滚动列表承载卡片。
- **FR-010**: “分区预报”模块 MUST 使用外层圆角矩形容器包裹。
- **FR-011**: “分区预报”模块 MUST 按 `DDATETIME` 仅展示今天及今天之后的预报记录。
- **FR-012**: “分区预报”模块中每个预报时次 MUST 以独立圆角卡片展示三行信息，并按顺序居中显示 `WEATHERSTATUS`、`MINTEMPERATURE/MAXTEMPERATURE`、`DDATETIME`。
- **FR-013**: 当“分区预报”卡片数量超过可视宽度时，模块 MUST 使用横向滚动列表承载卡片。
- **FR-014**: “日月时刻”模块 MUST 使用外层圆角矩形容器包裹，并按 `DDATETIME` 仅展示今天的数据。
- **FR-015**: “日月时刻”模块 MUST 将每个 `ATTRIBNAME` 与对应 `ATTRIBVALUE` 独立为一行，且行间使用分隔线分隔。
- **FR-016**: “灾害预警”模块 MUST 使用外层圆角矩形容器包裹，并按 `ISSUETIME` 与 `ISSUESTATE` 仅保留当前北京时间仍处于“发布”且未被“取消”的预警记录。
- **FR-017**: “灾害预警”模块 MUST 按返回的 `SIGNALTYPE` 顺序纵向逐行展示预警文本，格式为“序号 SIGNALTYPE SIGNALLEVEL预警”，其中 `SIGNALTYPE` 与 `SIGNALLEVEL` 间不插入空格。
- **FR-018**: “灾害预警”每行文本颜色 MUST 与 `SIGNALLEVEL` 对应；当 `SIGNALTYPE` 存在且 `SIGNALLEVEL` 为空时，默认使用黄色文本。
- **FR-019**: 用户 MUST 能点击每条有效预警文本打开详情弹窗，详情至少显示 `ISSUECONTENT` 与 `DISTRICT`。
- **FR-020**: 当 `SIGNALTYPE` 与 `SIGNALLEVEL` 均为空时，系统 MUST 仅显示 `gray-400` 文本“当前无生效信号。”，且不展示可点击详情入口。
- **FR-021**: 本功能新增的用户可见文案 MUST 使用简体中文，字段值本身按上游返回原样展示。
- **FR-022**: 系统 MUST 保持现有“请求时间”和“刷新结果”能力继续可用，不因新增按钮与弹窗而改变原有交互。

### API, UI, and Visualization Contract *(mandatory for web features)*

- **API-001**: 系统需要提供一个由“天气与观鸟指数”组件调用的 Route Handler，用于按 `parkId` 返回预报预警弹窗所需的四类数据（分区逐时预报、分区预报、日月时刻、灾害预警）；该接口为只读查询，支持按公园所属区县聚合数据。
- **API-002**: 该 Route Handler 的成功响应 MUST 同时包含四个模块的独立数据块与各自状态字段，允许某一模块失败时其他模块仍返回成功数据。
- **API-003**: 该 Route Handler 的失败响应 MUST 至少区分“公园参数无效”“上游服务不可用”“区县无可用数据”，并返回可直接用于前端展示的简体中文错误信息。
- **API-004**: 灾害预警响应数据 MUST 保留 `ISSUETIME`、`ISSUESTATE`、`SIGNALTYPE`、`SIGNALLEVEL`、`ISSUECONTENT`、`DISTRICT` 字段，以支持过滤、着色、列表文案和详情弹窗。
- **UI-001**: 现有“天气与观鸟指数”客户端组件需要新增一个深绿色入口按钮和一个顶层滚动弹窗容器；弹窗内部包含四个独立模块容器与模块标题。
- **UI-002**: 弹窗中的“分区逐时预报”和“分区预报”模块需要采用可横向滚动的卡片列表布局；每张卡片纵向三行居中展示指定字段。
- **UI-003**: 弹窗中的“日月时刻”模块需要采用逐行键值布局并使用分隔线；“灾害预警”模块需要采用可点击的纵向文本列表和详情弹窗。
- **UI-004**: 所有新增状态（加载、空状态、错误、无生效信号）必须在手机竖屏下可读，并保持按钮、弹窗与卡片的圆角矩形视觉一致性。
- **UI-005**: 新增中文文案至少包括“点击获取预报预警”“分区逐时预报”“分区预报”“日月时刻”“灾害预警”“当前无生效信号。”及各模块加载/失败提示。
- **CHART-001**: `N/A`。本功能不新增 Apache ECharts 图表。

### Key Entities *(include if feature involves data)*

- **Forecast Warning Dialog Context**: 弹窗查询上下文，包含当前 `parkId`、区县标识、请求时间（北京时间）和四模块独立状态。
- **Hourly District Forecast Item**: 分区逐时预报条目，包含 `DDATETIME`、`WEATHERSTATUS`、`QPFTEMP` 及可展示状态。
- **District Forecast Item**: 分区预报条目，包含 `DDATETIME`、`WEATHERSTATUS`、`MINTEMPERATURE`、`MAXTEMPERATURE`。
- **SunMoon Timing Item**: 日月时刻条目，包含 `DDATETIME`、`ATTRIBNAME`、`ATTRIBVALUE`。
- **Disaster Warning Item**: 灾害预警条目，包含 `ISSUETIME`、`ISSUESTATE`、`SIGNALTYPE`、`SIGNALLEVEL`、`ISSUECONTENT`、`DISTRICT` 及生效判定结果。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% 的用户可在进入“分析”页后 5 秒内定位并点击“点击获取预报预警”按钮打开弹窗。
- **SC-002**: 在网络正常场景下，90% 的弹窗请求在 5 秒内返回至少 3 个模块的可展示结果。
- **SC-003**: 100% 的“分区逐时预报”和“分区预报”卡片都按三行字段顺序正确展示，且在数据超宽场景下可横向滚动浏览全部卡片。
- **SC-004**: 100% 的“灾害预警”列表渲染符合颜色与文案规则：级别缺失默认黄色、类型与级别均空时显示“当前无生效信号。”灰色文本。
- **SC-005**: 在 375px-430px 竖屏宽度范围内，弹窗整体可纵向滚动，除两类卡片列表外不出现非预期横向溢出。

## Assumptions

- “公园地图”已持续提供当前选中公园与所属区县信息，预报预警功能直接复用该上下文。
- 四类数据源均可通过项目 Route Handler 统一访问，前端不直接请求外部上游接口。
- 当前北京时间以服务端统一时间基准计算，用于 `DDATETIME`、`ISSUETIME` 的筛选判断。
- 灾害预警的“生效”定义以“发布且未取消”为准，不额外引入新的业务状态枚举。
- 若某模块暂无有效数据，系统通过模块级空状态提示处理，不以伪造默认值替代真实返回。
