# Feature Specification: 登录/注册入口与弹窗

**Feature Branch**: `006-login-register`  
**Created**: 2026-04-01  
**Status**: Draft  
**Input**: User description: "我要给整个网页新增一个“登录/注册”功能。将登录、注册入口以适当大小放置在固定横栏下方，页面右上角。点击登录或注册后弹出登录/注册弹窗。登录需要“助手账号”和“用户密码”；注册暂时不开放接口，仅提示：一个默认管理员头像图片在上方，下方是文字“请联系管理员注册与登录”。添加一个测试账号：助手账号=WTBTEST，密码=123456."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 使用测试账号登录 (Priority: P1)

作为首次进入网页的用户，我希望在页面右上角快速看到“登录/注册”入口，并能通过弹窗输入助手账号和用户密码完成登录，这样我可以在不离开当前页面的情况下进入受控使用状态。

**Why this priority**: 登录是这个功能的最核心能力；如果用户无法进入登录流程，后续注册提示和页面身份状态都没有意义。

**Independent Test**: 打开任意页面，确认固定横栏下方右上角出现登录和注册入口；点击“登录”后输入测试账号 `WTBTEST` 和密码 `123456`，确认登录成功且弹窗关闭。

**Acceptance Scenarios**:

1. **Given** 用户尚未登录，**When** 页面完成首屏渲染，**Then** 固定横栏下方右上角显示尺寸适中的“登录”和“注册”入口。
2. **Given** 用户点击“登录”入口，**When** 登录弹窗打开，**Then** 弹窗中显示“助手账号”和“用户密码”两个输入项以及提交操作。
3. **Given** 用户在登录弹窗中输入 `WTBTEST` 和 `123456`，**When** 用户提交登录，**Then** 系统判定登录成功、关闭弹窗，并将页面身份状态更新为已登录。
4. **Given** 用户输入错误的助手账号或密码，**When** 用户提交登录，**Then** 弹窗保持打开并展示明确的简体中文错误提示。

---

### User Story 2 - 查看注册引导提示 (Priority: P2)

作为尚未拥有账号的用户，我希望点击“注册”后看到清晰的管理员联系指引，而不是进入一个不可用的注册表单，这样我能立刻知道下一步该怎么做。

**Why this priority**: 当前注册接口明确不开放，如果没有清晰的替代指引，用户会误以为页面故障或流程缺失。

**Independent Test**: 点击“注册”入口，确认弹窗显示默认管理员头像图片，并在下方展示“请联系管理员注册与登录”文字，不出现可提交的注册表单。

**Acceptance Scenarios**:

1. **Given** 用户尚未登录，**When** 用户点击“注册”入口，**Then** 页面弹出注册引导弹窗。
2. **Given** 注册引导弹窗已打开，**When** 用户查看弹窗主体，**Then** 顶部显示默认管理员头像图片，下方显示“请联系管理员注册与登录”文字。
3. **Given** 当前阶段未开放注册接口，**When** 用户打开注册引导弹窗，**Then** 页面不得展示可提交的注册表单或制造“可直接注册”的误导性提示。

---

### User Story 3 - 在当前会话中识别登录状态 (Priority: P3)

作为已经完成登录的用户，我希望页面右上角的身份入口能反映当前登录状态，这样我能确认自己已经通过验证，而不是反复重复登录。

**Why this priority**: 登录成功如果没有可见反馈，用户无法判断是否已经进入有效状态，容易重复操作并降低信任感。

**Independent Test**: 使用测试账号完成登录后，确认右上角身份入口不再保持初始未登录展示，而会显示当前已登录状态和助手账号标识。

**Acceptance Scenarios**:

1. **Given** 用户已使用有效测试账号登录成功，**When** 页面恢复正常浏览状态，**Then** 右上角身份区域显示已登录状态与当前助手账号标识。
2. **Given** 用户已经处于已登录状态，**When** 用户继续切换“分析”“识别”“记录”页面，**Then** 当前会话中的登录状态保持一致，不因页面切换而立刻丢失。
3. **Given** 用户刷新页面或重新打开网页，**When** 本期未配置持久化登录能力，**Then** 系统允许回到默认未登录状态，并要求用户重新登录。

### Edge Cases

- 用户点击“登录”或“注册”入口时，如果已有另一个身份弹窗打开，系统如何避免同时出现多个重叠弹窗？
- 用户提交登录时任一字段为空，页面如何阻止提交并给出清晰提示？
- 用户输入正确账号但错误密码时，系统如何明确提示失败而不是静默无响应？
- 默认管理员头像图片资源暂时不可用时，注册引导弹窗如何避免出现破损占位？
- 用户在登录成功后立即切换页面或重复点击入口，系统如何避免状态闪烁或重复提交？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在整个网页共享的固定横栏下方、页面右上角新增“登录”和“注册”入口。
- **FR-002**: 系统 MUST 使登录/注册入口在目标手机竖屏宽度内清晰可点，且不得遮挡固定横栏或首屏主要内容。
- **FR-003**: 用户 MUST 能够点击“登录”入口打开登录弹窗。
- **FR-004**: 登录弹窗 MUST 提供“助手账号”和“用户密码”两个输入项，并使用简体中文标签与提示文案。
- **FR-005**: 系统 MUST 提供一个内置测试账号，其助手账号为 `WTBTEST`，用户密码为 `123456`。
- **FR-006**: 当用户提交与测试账号完全匹配的登录信息时，系统 MUST 判定登录成功并关闭登录弹窗。
- **FR-007**: 当用户提交错误账号、错误密码或空字段时，系统 MUST 阻止登录并展示明确的简体中文反馈。
- **FR-008**: 用户 MUST 能够点击“注册”入口打开注册引导弹窗。
- **FR-009**: 注册引导弹窗 MUST 在顶部展示一个默认管理员头像图片，并在下方展示“请联系管理员注册与登录”文字。
- **FR-010**: 当前阶段系统 MUST NOT 提供可提交的注册接口、注册表单或注册成功流程。
- **FR-011**: 登录/注册弹窗 MUST 位于当前页面图层上方，打开时不得被顶部固定横栏或其他悬浮元素遮挡。
- **FR-012**: 登录成功后，系统 MUST 在当前会话内将右上角身份区域更新为已登录状态，并显示当前助手账号标识。
- **FR-013**: 当前会话内的登录状态 MUST 在“分析”“识别”“记录”页面切换过程中保持一致。
- **FR-014**: 在当前阶段未引入持久化登录能力时，系统 MAY 在页面刷新后恢复为默认未登录状态，但必须保持行为一致且可预期。
- **FR-015**: 登录/注册相关的所有用户可见文字、错误提示、按钮标签和占位说明 MUST 使用简体中文。
- **FR-016**: 该功能 MUST 保持在当前移动网页内部完成，不新增新的顶层页面或跳转到外部注册系统。

### API, UI, and Visualization Contract *(mandatory for web features)*

- **API-001**: 需要一个身份校验用 Route Handler：`POST /api/auth/login`。调用方为网页右上角的登录弹窗。请求体 MUST 至少包含 `assistantAccount` 与 `userPassword`；成功响应 MUST 至少返回登录结果、助手账号显示值与用户提示；失败响应 MUST 至少区分空字段与账号密码错误两类情况。
- **API-002**: 当前阶段注册流程不开放后端接口；注册引导弹窗使用静态说明内容，不调用注册创建接口。
- **UI-001**: 需要新增一个全局身份入口区域，放置在固定横栏下方、页面右上角，且在“分析”“识别”“记录”三个页面中保持一致。
- **UI-002**: 需要一个登录弹窗，使用 Tailwind CSS + shadcn/ui 组合实现，至少包含标题、两个输入项、提交按钮、关闭能力、加载态和错误提示区域。
- **UI-003**: 需要一个注册引导弹窗，使用 Tailwind CSS + shadcn/ui 组合实现，至少包含标题、默认管理员头像图片、联系提示文案和关闭能力。
- **UI-004**: 登录成功后，右上角身份区域 MUST 从未登录入口状态切换为已登录状态，并展示当前助手账号标识。
- **UI-005**: 登录弹窗与注册引导弹窗 MUST 使用统一的移动端弹层样式，支持遮罩关闭与显式关闭，不得导致页面出现横向滚动。
- **UI-006**: 登录入口、注册入口、弹窗标题、输入标签、按钮、错误提示、注册引导文案和已登录状态文案 MUST 全部使用简体中文。
- **CHART-001**: `N/A`。本功能不包含图表或数据可视化需求。

### Key Entities *(include if feature involves data)*

- **Auth Entry State**: 网页右上角身份入口的展示状态，区分未登录入口状态与已登录展示状态。
- **Login Credential**: 登录提交时使用的一组身份凭据，包含助手账号和用户密码。
- **Test Account**: 当前阶段用于校验登录流程的内置账号，固定为 `WTBTEST / 123456`。
- **Registration Notice**: 注册引导弹窗中展示的静态管理员头像与联系说明内容。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% 的首屏访问中，用户在进入任一页面后 2 秒内可以看到固定横栏下方右上角的“登录/注册”入口。
- **SC-002**: 95% 的正常交互中，用户点击“登录”或“注册”后 1 秒内可以看到对应弹窗。
- **SC-003**: 对于输入 `WTBTEST / 123456` 的登录尝试，100% 的有效提交都能返回成功结果并关闭登录弹窗。
- **SC-004**: 对于空字段或错误凭据的登录尝试，100% 的提交都能显示明确的失败反馈，而不是无响应或页面跳转。
- **SC-005**: 在目标手机竖屏宽度范围内，登录/注册入口、弹窗内容和已登录状态都不引入横向滚动，且不遮挡固定横栏。

## Assumptions

- 当前阶段的“登录/注册”目标是补齐移动网页的基础身份入口与演示登录能力，而不是接入完整账号体系。
- 本期只提供一个内置测试账号 `WTBTEST / 123456`，不包含多账号管理、密码修改、找回密码或持久化会话能力。
- 注册接口在本期明确关闭，因此注册入口只承担引导说明作用，不需要创建账户或提交审核。
- 默认管理员头像图片将作为项目内可访问资源提供；若资源缺失，界面仍需保持结构完整并可展示联系说明。
- 所有面向前端的登录校验能力都通过 App Router Route Handlers 暴露，前端不会直接在页面组件中硬编码成功或失败流程。
