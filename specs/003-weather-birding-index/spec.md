# Feature Specification: Analysis Weather and Birding Index

**Feature Branch**: `003-weather-birding-index`  
**Created**: 2026-03-31  
**Status**: Draft  
**Input**: User description: "我要在“分析”页面继续添加天气展示和观鸟指数判断功能。区域布局在地图下方，适配网页宽度。功能：根据在地图模块选择的公园所在的市辖区（县级），查询天气信息。将查询到的天气信息发给AI，让AI判断今天的观鸟指数（观鸟指数分为“适宜”、“较适宜”、“不适宜”）并返回。将观鸟指数和所有查询到的天气信息展示在地图下方。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 查看当前公园的天气与观鸟指数 (Priority: P1)

作为访问分析页的用户，我希望在地图下方直接看到当前所选公园对应区县的天气信息和今天的观鸟指数，这样我不用离开分析页就能判断今天是否适合观鸟。

**Why this priority**: 这是该需求的核心价值。如果用户不能在分析页直接获得天气与观鸟指数，这个功能就没有形成独立价值。

**Independent Test**: 打开分析页并保持默认公园选中状态，确认地图下方会展示该公园所属区县的天气信息，以及一个仅能取值为“适宜”“较适宜”“不适宜”的今日观鸟指数。

**Acceptance Scenarios**:

1. **Given** 用户首次进入分析页且地图模块已经有默认公园，**When** 页面完成首轮数据加载，**Then** 地图下方显示默认公园所属区县的天气信息和今日观鸟指数。
2. **Given** 天气与观鸟指数数据成功返回，**When** 用户查看地图下方内容区域，**Then** 用户可以同时看到区县名称、今日天气信息集合和一个明确的观鸟指数结果。
3. **Given** 用户在目标手机宽度范围内访问分析页，**When** 天气与观鸟指数区域渲染完成，**Then** 该区域横向适配页面宽度且不引入水平滚动。

---

### User Story 2 - 切换公园后同步刷新结果 (Priority: P2)

作为正在比较不同公园的用户，我希望在地图模块切换公园后，地图下方的天气信息和观鸟指数自动跟随刷新，这样我可以快速比较不同区县当天的观鸟条件。

**Why this priority**: 公园切换后的联动刷新决定了该功能是否真正服务于地图选择行为，而不是静态展示一份与当前分析对象无关的信息。

**Independent Test**: 在分析页依次切换多个预设公园，确认每次切换后地图下方展示的区县、天气信息和观鸟指数都更新为当前公园对应结果，且页面无刷新。

**Acceptance Scenarios**:

1. **Given** 用户位于分析页且地图模块可切换公园，**When** 用户选择新的公园，**Then** 地图下方的天气与观鸟指数区域自动刷新为新公园所属区县的数据。
2. **Given** 用户从一个公园切换到另一个公园，**When** 新结果加载完成，**Then** 页面展示的区县名称、天气信息和观鸟指数必须与当前选中的公园一致。
3. **Given** 用户切换公园，**When** 地图下方区域更新，**Then** 页面不发生整页刷新，且用户仍停留在分析页上下文中。

---

### User Story 3 - 在异常与快速切换下保持可理解结果 (Priority: P3)

作为分析页用户，我希望在天气服务波动、AI 判断失败或快速切换多个公园时，页面仍能给出清晰、可信的反馈，这样我不会因为旧结果、空白区域或错误状态而误判当天是否适合观鸟。

**Why this priority**: 天气查询和 AI 判断都依赖外部能力，若没有明确的异常回退与最新选择一致性，用户很容易看到过期或误导性的结论。

**Independent Test**: 模拟天气查询失败、AI 判断失败和快速连续切换公园，确认页面要么显示与最后一次选择一致的结果，要么显示清晰的不可用提示，但不会把旧公园的结果误当作当前结果展示。

**Acceptance Scenarios**:

1. **Given** 用户快速连续切换多个公园，**When** 多次请求先后返回，**Then** 页面最终只保留与最后一次公园选择一致的天气信息和观鸟指数。
2. **Given** 天气信息未能成功获取，**When** 地图下方区域进入失败状态，**Then** 页面显示清晰的天气不可用提示，且不会继续显示误导性的当前观鸟指数结果。
3. **Given** 天气信息已成功获取但 AI 判断失败，**When** 页面展示结果，**Then** 用户仍可看到天气信息，同时看到明确的观鸟指数不可用状态。

### Edge Cases

- 选中的公园无法映射到有效的区县时，系统如何提示并阻止错误查询？
- 天气信息只返回部分字段时，页面如何继续展示已获得的数据并标明缺失项？
- 天气查询成功但 AI 返回了不在“适宜”“较适宜”“不适宜”范围内的结果时，系统如何处理异常返回值？
- 用户在前一个公园请求尚未完成时切换到新公园，系统如何避免旧请求覆盖新选择的结果？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在分析页地图模块下方提供一个专门的“天气与观鸟指数”展示区域。
- **FR-002**: 系统 MUST 以地图模块当前选中的公园作为唯一分析上下文，并据此确定对应的市辖区或县级行政区。
- **FR-003**: 系统 MUST 在分析页默认公园加载完成后，自动获取该公园对应区县的今天天气信息。
- **FR-004**: 系统 MUST 将获取到的天气信息发送给 AI，以生成当天观鸟指数判断结果。
- **FR-005**: 观鸟指数结果 MUST 严格限制为“适宜”“较适宜”“不适宜”三种之一。
- **FR-006**: 系统 MUST 在地图下方同时展示以下天气信息集合：区县名称、天气现象或天气概述、温度信息、湿度信息、风向或风力信息、降水相关信息、空气质量相关信息，以及天气数据更新时间。
- **FR-007**: 系统 MUST 在天气信息区域中展示当天观鸟指数结果，并将其与当前所示天气信息保持对应关系。
- **FR-008**: 用户切换地图中的公园后，系统 MUST 自动重新查询新公园对应区县的天气，并重新生成当天观鸟指数。
- **FR-009**: 前端 MUST 仅通过项目批准的后端接口获取天气与观鸟指数数据，不得直接从浏览器调用天气服务或 AI 服务。
- **FR-010**: 当天气与观鸟指数正在刷新时，系统 MUST 在地图下方展示明确的加载状态，且不得把旧公园结果误标为当前结果。
- **FR-011**: 当天气查询失败时，系统 MUST 显示清晰的失败提示，并停止展示基于失败查询生成的观鸟指数结果。
- **FR-012**: 当天气查询成功但 AI 判断失败时，系统 MUST 保留已查询到的天气信息展示，并将观鸟指数标记为暂不可用。
- **FR-013**: 当用户快速连续切换多个公园时，系统 MUST 保证最终展示结果与最后一次选择的公园一致。
- **FR-014**: 天气与观鸟指数展示区域 MUST 适配分析页可用宽度，并在目标手机宽度范围内不引入水平滚动。
- **FR-015**: 该功能 MUST 保持在分析页内部完成，不新增新的顶层导航页面或脱离地图上下文的独立流程。

### API, UI, and Visualization Contract *(mandatory for web features)*

- **API-001**: 需要一个聚合型 Route Handler：`GET /api/analysis/birding-outlook?parkId={parkId}`。调用方为分析页地图下方的天气与观鸟指数展示区域。成功返回应包含当前 `parkId`、公园名称、区县信息、天气信息对象、观鸟指数对象，以及各自的可用状态；失败返回应至少区分无效公园、天气不可用和系统异常三类错误。
- **API-002**: 成功响应中的天气信息对象 MUST 至少覆盖区县名称、天气概述、温度、湿度、风、降水、空气质量、更新时间；观鸟指数对象 MUST 至少覆盖可用状态和指数等级，指数等级仅允许“适宜”“较适宜”“不适宜”。
- **API-003**: 当天气可用但观鸟指数暂不可用时，接口 SHOULD 返回部分成功结果，使前端仍能渲染天气信息并显式标记观鸟指数不可用，而不是把整个区域视为完全失败。
- **UI-001**: 分析页需要在现有地图模块下方新增一个天气与观鸟指数内容区，该内容区由 Tailwind CSS + shadcn/ui 组合实现，至少包含信息容器卡片、观鸟指数状态展示、天气字段列表、分隔结构，以及可选的重试操作入口。
- **UI-002**: 该内容区 MUST 与当前地图选择状态联动，并在视觉上明确区分加载中、数据成功、部分成功、完全失败四种状态。
- **UI-003**: 当数据成功返回时，观鸟指数必须在信息层级上突出展示，天气字段必须作为可扫描的明细列表出现在同一区域，且整体布局在手机竖屏宽度内易读。
- **UI-004**: 当结果不可用或部分不可用时，页面 MUST 使用一致的轻量提示文案说明当前状态，且不遮挡地图模块本身的选择能力。
- **CHART-001**: `N/A`。本功能展示的是天气信息和 AI 判断结果，不包含图表可视化需求。

### Key Entities *(include if feature involves data)*

- **Park Weather Context**: 当前分析页选中的公园上下文，包含公园标识、公园名称、所属城市和所属区县，用于驱动天气查询和观鸟指数判断。
- **District Weather Snapshot**: 某个区县当天的天气快照，包含用户可见的天气字段集合及其更新时间，用于展示和 AI 评估。
- **Birding Index Assessment**: 基于天气快照生成的当日观鸟指数结果，包含可用状态、指数等级和生成时间，用于帮助用户判断当天是否适合观鸟。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% 的正常网络访问中，用户在进入分析页后 3 秒内可以在地图下方看到默认公园对应区县的天气信息和今日观鸟指数。
- **SC-002**: 95% 的公园切换操作中，地图下方结果会在 3 秒内更新为当前公园对应区县的天气信息和观鸟指数，且无需整页刷新。
- **SC-003**: 100% 的成功判断结果都只会显示“适宜”“较适宜”“不适宜”三种观鸟指数之一。
- **SC-004**: 在目标手机宽度范围内，天气与观鸟指数区域在成功、加载、部分成功、失败四种状态下都不需要水平滚动。
- **SC-005**: 在快速连续切换公园的验证场景中，最终显示的公园、区县、天气信息和观鸟指数与最后一次选择保持一致的比例达到 100%。

## Assumptions

- 本功能仅面向当前分析页已有的预设公园选择能力，不包含用户自定义地点输入或新的地点搜索流程。
- 每个预设公园在本期范围内都可以稳定映射到唯一的市辖区或县级行政区，用于天气查询。
- 本期“今天天气信息”指当日可供即时判断的天气快照，不扩展为多日预测或逐小时天气趋势。
- 本期观鸟指数仅基于当次查询得到的天气信息进行 AI 判断，不引入历史观测记录、迁徙数据或个性化偏好参与计算。
- 所有面向前端的数据访问都将通过 App Router Route Handlers 暴露，前端不会直接连接天气服务或 AI 服务。
