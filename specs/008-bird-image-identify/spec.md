# Feature Specification: 识别页鸟影识别

**Feature Branch**: `008-bird-image-identify`  
**Created**: 2026-04-02  
**Status**: Draft  
**Input**: User description: "我要在“识别”页面添加功能。“工具介绍”改为“鸟影识别”,“鸟类识别工具”标题不变。1. 图片上传展示框 布局：标题下方，横向展示，比例4:3。功能：默认状态：空白，中间显示 “点击上传图片” 提示文字，文字上方配相机图标，提示文本与相机图标居中。交互：点击区域可从本地上传图片，上传后立即在框内显示缩略图（保持比例，不拉伸）。2. 鸟类识别结果 布局：图片展示框下方展示。功能：调用千问3.6plus模型，识别鸟类并返回。显示内容：识别到的鸟种名称：简体中文学名、英文学名、拉丁文学名。交互：上传图片后居中显示 “识别中......” 加载动画，识别完成后展示结果。3. 鸟类百科简介 布局：识别结果下方整行展示。功能：根据识别结果，询问千问具体信息。展示内容：干净的文本简介，包括物种特征、生活习性、分布区域、保护级别（包括国内保护、世界濒危等级，括号内内容不展示）等。4.底部浅色居中小字提示：识别结果仅供参考。鸟类图片越接近标准照，准确率越高。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 上传图片并查看鸟类识别结果 (Priority: P1)

作为在移动端使用识别页的观鸟用户，我希望直接在页面标题下方上传一张本地鸟类图片，并在同一页面中看到缩略图和识别结果，这样我可以快速确认图片里最可能的鸟种而不用跳转到其他页面。

**Why this priority**: 图片上传与识别结果是识别页的核心价值。如果用户不能在当前页面完成上传并得到鸟种结果，后续百科简介和提示文案都失去意义。

**Independent Test**: 打开“识别”页面，确认标题前导文案改为“鸟影识别”；点击上传框选择一张本地图片后，确认图片立即预览，页面显示“识别中......”，随后在结果区展示中文名、英文名和拉丁学名。

**Acceptance Scenarios**:

1. **Given** 用户进入“识别”页面且尚未上传图片，**When** 页面完成首屏渲染，**Then** 标题上方的小字文案显示为“鸟影识别”，主标题仍显示“鸟类识别工具”，标题下方出现一个横向 4:3 的图片上传展示框。
2. **Given** 上传展示框处于默认空状态，**When** 用户查看该区域，**Then** 框内居中显示相机图标，图标下方显示“点击上传图片”，且二者保持垂直居中排列。
3. **Given** 用户点击上传展示框并从本地选择了一张有效图片，**When** 选择完成，**Then** 页面立即在展示框内显示该图片缩略图，并保持原始比例、不拉伸、不超出展示框。
4. **Given** 用户刚上传一张有效图片，**When** 系统开始处理识别，**Then** 图片展示框下方居中显示“识别中......”加载状态，直到识别结果可用。
5. **Given** 识别完成且返回了有效鸟种信息，**When** 结果区渲染完成，**Then** 页面在图片展示框下方展示该鸟种的简体中文名、英文学名和拉丁学名。

---

### User Story 2 - 查看与识别结果对应的鸟类百科简介 (Priority: P2)

作为已经拿到识别结果的观鸟用户，我希望在识别结果下方继续看到一段整洁、易读的鸟类百科简介，这样我能立刻理解该鸟种的主要特征、习性、分布和保护等级。

**Why this priority**: 识别结果只告诉用户“它可能是什么”，百科简介进一步解释“为什么是它、它有什么特点”，能显著提升识别页的参考价值。

**Independent Test**: 上传一张能识别出明确鸟种的图片，确认识别结果下方出现整行百科简介，且内容至少覆盖物种特征、生活习性、分布区域和保护级别，保护级别中的括号内容不显示。

**Acceptance Scenarios**:

1. **Given** 系统已经得到一条有效的鸟类识别结果，**When** 页面继续完成对应信息展示，**Then** 识别结果下方出现一整行鸟类百科简介区域。
2. **Given** 百科简介区域成功展示，**When** 用户阅读内容，**Then** 简介中至少包含物种特征、生活习性、分布区域和保护级别四类信息。
3. **Given** 百科简介中包含保护级别相关信息，**When** 系统整理最终展示文本，**Then** 保护级别仅显示清理后的可读内容，不展示括号内的附加说明。
4. **Given** 用户重新上传了另一张图片，**When** 新一轮识别开始，**Then** 上一张图片对应的百科简介不再继续作为当前图片的结果展示。

---

### User Story 3 - 在识别失败或更换图片时保持页面清晰可用 (Priority: P3)

作为移动端识别页用户，我希望即使上传了无效图片、识别结果不明确，或者想要重新选择一张图片，页面也能保持当前布局清晰、提示明确，并允许我立刻重试。

**Why this priority**: 识别能力依赖用户上传图片质量和外部识别服务，失败与替换图片是高频边界场景；如果页面不能明确区分这些状态，用户会误以为当前结果仍然有效。

**Independent Test**: 依次测试无效图片、无法识别图片、上传新图片覆盖旧结果等场景，确认页面始终保留上传入口、给出明确简体中文提示，并且不会将旧图片结果错误映射到新图片上。

**Acceptance Scenarios**:

1. **Given** 用户上传了一张图片但图片中未包含可识别的鸟类，**When** 系统完成识别判断，**Then** 页面直接提示“图片中未包含可识别的鸟类！”，并保留重新上传图片的入口。
2. **Given** 用户已得到一组识别结果，**When** 用户再次点击上传展示框并选择另一张图片，**Then** 页面立即以新图片替换旧缩略图，并清除旧图片对应的识别结果与百科简介上下文。
3. **Given** 识别页出现失败提示或空结果，**When** 用户继续停留在当前页面，**Then** 页面底部仍显示“识别结果仅供参考。鸟类图片越接近标准照，准确率越高。”提示文案。
4. **Given** 用户处于手机竖屏浏览状态，**When** 页面在空状态、识别中、成功或失败之间切换，**Then** 页面始终可上下滚动且不出现横向滚动或横屏布局。

### Edge Cases

- 用户上传了非图片文件、损坏图片或无法读取的本地文件时，页面如何阻止无效识别并保留重新上传入口？
- 用户上传的图片中没有鸟类主体、主体过小，或出现多只鸟类而无法稳定识别时，系统如何统一返回“图片中未包含可识别的鸟类！”而不是给出误导性的确定性结论？
- 用户在上一轮识别尚未完成时再次上传新图片，系统如何确保最终只展示最新一张图片对应的识别结果和百科简介？
- 识别成功但百科简介整理失败时，页面如何保留已得到的鸟种名称并单独提示百科内容暂不可用？
- 用户连续多次更换图片时，页面如何确保每次都用最新上传内容替换旧缩略图与旧结果，而不产生残留文本？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将“识别”页面标题上方的小字文案由“工具介绍”改为“鸟影识别”，并保持主标题“鸟类识别工具”不变。
- **FR-002**: 系统 MUST 在标题区域下方提供一个横向图片上传展示框，并保持该展示框的视觉比例为 4:3。
- **FR-003**: 系统 MUST 在图片尚未上传时，将上传展示框显示为默认空状态，并在框内居中展示相机图标与“点击上传图片”提示文字。
- **FR-004**: 系统 MUST 允许用户通过点击上传展示框从本地选择单张图片进行上传。
- **FR-005**: 系统 MUST 在用户选择图片后立即于展示框内显示该图片缩略图，并保持原图比例、不拉伸、不溢出。
- **FR-006**: 系统 MUST 在每次成功选择新图片后，将其视为当前唯一有效待识别图片，并用它替换上一张图片及其相关展示上下文。
- **FR-007**: 系统 MUST 在图片进入识别流程后，于图片展示框下方居中显示“识别中......”加载状态，直到本轮识别得到可展示结果或明确失败。
- **FR-008**: 系统 MUST 在识别完成后展示鸟类识别结果，且识别结果至少包含简体中文名、英文学名和拉丁学名。
- **FR-009**: 系统 MUST 将鸟类识别结果展示在图片展示框下方，并保持其与上传区域同一内容宽度范围内可读。
- **FR-010**: 系统 MUST 在识别结果下方展示一整行鸟类百科简介，并将该区域视为当前识别结果的补充说明。
- **FR-011**: 系统 MUST 在百科简介中至少展示物种特征、生活习性、分布区域和保护级别信息。
- **FR-012**: 系统 MUST 在展示保护级别信息时移除括号内内容，仅保留清理后的可读文本。
- **FR-013**: 系统 MUST 在识别页底部展示浅色、居中、小字提示“识别结果仅供参考。鸟类图片越接近标准照，准确率越高。”
- **FR-014**: 当图片中未包含可识别的鸟类，或系统无法给出有效鸟种结论时，系统 MUST 直接向用户显示“图片中未包含可识别的鸟类！”，并允许用户继续重新上传图片。
- **FR-015**: 当百科简介无法生成但识别结果已可用时，系统 MUST 保留识别结果展示，并对百科简介区域单独给出明确提示，而不是一并清空整页结果。
- **FR-016**: 系统 MUST 确保页面只展示与最新上传图片对应的识别结果和百科简介，不得将旧图片结果残留到新图片上下文中。
- **FR-017**: “识别”页面与本功能相关的标题、提示、加载、失败、空状态和底部说明文案 MUST 全部使用简体中文；英文学名和拉丁学名作为识别结果内容例外。
- **FR-018**: 系统 MUST 保持该功能适配手机竖屏宽度，并在空状态、识别中、成功与失败状态下都不引入横向滚动或横屏布局。

### API, UI, and Visualization Contract *(mandatory for web features)*

- **API-001**: 系统需要提供一个由“识别”页面调用的上传识别 Route Handler，用于接收用户选择的单张本地图片并返回规范化的鸟类识别结果；成功响应 MUST 至少包含当前图片对应的中文名、英文学名、拉丁学名以及结果可用状态；失败响应 MUST 至少区分图片无效、图片中未包含可识别鸟类和服务暂不可用，其中“未包含可识别鸟类”场景需返回固定提示“图片中未包含可识别的鸟类！”。
- **API-002**: 系统需要提供与识别结果配套的百科信息返回能力；该能力可以与上传识别合并在同一次后端响应中，也可以通过同一识别流程的后续服务端步骤返回，但前端 MUST 仅通过项目允许的 Route Handler 获取该简介内容，而不是直接连接外部模型服务。
- **API-003**: 当识别结果可用但百科简介不可用时，接口响应 MUST 支持前端同时呈现“识别结果成功 + 百科简介失败”的部分成功状态。
- **UI-001**: “识别”页面需要在现有标题区域下方新增图片上传展示框、识别结果区和百科简介区，并使用 Tailwind CSS + shadcn/ui 风格组件组合实现卡片、占位、加载和提示状态。
- **UI-002**: 图片上传展示框 MUST 支持默认空状态、上传后预览状态、识别中状态和失败后可重试状态，并保持同一布局结构在手机竖屏中稳定显示。
- **UI-003**: 识别结果区 MUST 以清晰分组方式展示中文名、英文学名和拉丁学名；百科简介区 MUST 位于识别结果区下方并保持整行可读文本布局。
- **UI-004**: 识别页用户可见文案 MUST 包含“鸟影识别”“点击上传图片”“识别中......”“图片中未包含可识别的鸟类！”以及“识别结果仅供参考。鸟类图片越接近标准照，准确率越高。”等简体中文文案。
- **CHART-001**: `N/A`。本功能不包含图表或数据可视化要求。

### Key Entities *(include if feature involves data)*

- **Uploaded Bird Image**: 用户在“识别”页面当前选中的单张本地图片，包含文件有效性、预览展示状态和最新上传上下文。
- **Bird Recognition Result**: 当前图片对应的鸟种识别结果，包含简体中文名、英文学名、拉丁学名以及结果可用性状态。
- **Bird Encyclopedia Summary**: 基于当前识别结果生成的整洁文本简介，包含物种特征、生活习性、分布区域和保护级别等摘要内容。
- **Identify Page Feedback State**: 识别页当前向用户显示的空状态、识别中状态、成功状态、部分成功状态和失败状态，用于约束页面始终只呈现最新上传图片的上下文。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% 的用户能够在进入“识别”页面后 10 秒内完成一张本地图片上传并看到对应缩略图。
- **SC-002**: 90% 的有效鸟类图片上传后，用户能够在 15 秒内看到至少包含中文名、英文学名和拉丁学名的识别结果。
- **SC-003**: 95% 的成功识别结果都能在同一页面继续展示对应百科简介，且简介至少覆盖物种特征、生活习性、分布区域和保护级别四类信息。
- **SC-004**: 100% 的重新上传场景中，页面最终只展示最新一张图片的缩略图、识别结果和百科简介，不出现旧结果残留。
- **SC-005**: 在目标手机竖屏宽度内，空状态、识别中、成功、部分成功和失败场景下均不需要横向滚动即可完成主要阅读与操作。

## Assumptions

- “识别”页面继续保留当前页面位置与导航入口，仅在页面内部补充鸟影识别能力，不新增新的顶层导航项。
- 每次识别仅处理用户当前选中的一张本地图片；用户重新选择图片后，上一张图片的结果不再作为当前上下文继续展示。
- 上传图片会先在本地生成预览，再由服务端识别流程处理；用户不需要手动确认第二次提交才能开始识别。
- 鸟类百科简介以当前识别结果为唯一来源，不要求用户额外填写鸟种名称或手动触发第二次查询。
- 当识别结果或百科简介不可用时，页面会保留当前上传入口与最新缩略图，方便用户立即重试。
- 本功能产生的用户可见文案默认使用简体中文，只有英文学名和拉丁学名作为识别结果内容保留原语言形式。
